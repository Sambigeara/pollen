Concepts

Pollen's surface is small because the model underneath is small. Five ideas are enough to explain everything else: a peer-to-peer mesh, a gossiped CRDT, locally-decided workload placement, a content-addressed store, and a capability model rooted in a single signing key.

The mesh

Every node has an ed25519 identity. Peers maintain at most one QUIC connection between them, multiplexed for gossip, service tunnels, and seed traffic. mTLS is end-to-end; there are no shared secrets and no separate ingress.

Finding each other

Two peers learn about each other through gossip. Once they've exchanged addresses they try a direct UDP punch. Most of the time it works, even between two NATs. When it doesn't, they fall back to relaying through any mutually-reachable cluster node.

Peer A and Peer B are both behind NAT, and want to talk. 1. Exchange candidate addresses (via gossip or relay): A's addrs ---> B B's addrs ---> A 2. Simultaneous UDP punch: A: SYN ---> B B: SYN ---> A 3a. If either SYN escapes its NAT, the connection is direct. 3b. Otherwise both peers route through a public node R that they can each reach. R is just a normal cluster node that happens to be publicly reachable: A <----> R <----> B

A relay is just a regular node that happens to be publicly reachable. Bootstrap one with pln bootstrap ssh root@host --admin on a machine with a public IP, or hint it on start with pln up --public; the mesh verifies actual reachability via NAT probing. Public nodes' addresses are woven into invite tokens automatically, so a joiner behind NAT has a route in without you plumbing anything.

One transport, one connection

QUIC carries everything. Gossip messages, service tunnels, and seed RPCs share the same multiplexed stream per peer. Encryption is end-to-end; relays cannot read traffic they forward. There is no separate VPN, ingress controller, or sidecar.

See the recipe for adding a relay and the bootstrap-ssh reference.

State and gossip

Every node holds the same document: a CRDT describing peers, services, workloads, sites, blobs, and the capability tree. CRDTs are designed so that any two nodes that have seen the same set of changes converge on identical state, regardless of the order they observed those changes.

That property gives Pollen three things for free:

  • No leader. No node is the source of truth. Every node can read, every admin-capable node can write, and every node observes the same world.
  • Partition tolerance. Split the cluster in half and both sides keep working. Reunite them and the document re-converges.
  • Local decisions. Because every node has the same view, every node can make placement and routing decisions without consulting anyone else.
$ pln seed ./hello.wasm Node A writes one CRDT change: c1 = workloads[hello] := { hash: 4f3a, ... } Gossip rounds (anti-entropy) ensure every other peer eventually receives c1 via at least one path. Once converged, every node sees the same: state.workloads[hello] = { hash: 4f3a, ... }

Changes ride QUIC streams between peers. Conflict resolution is built into the document itself: each entry is keyed by a stable identifier and most fields are write-once or last-writer-wins. Tombstones mark deletes and propagate the same way as creates, so a peer that's been offline for an hour will catch up on both sides of the timeline when it reconnects.

There's no "control plane" to fail because there's no control plane at all. Every node holds it.

Workloads (seeds)

A seed is a unit of code that the cluster runs. Today that's a WebAssembly module compiled to Extism's plugin shape, which you can produce from Go, Rust, JavaScript, Python, C#, or Zig. pln seed publishes one as a signed spec; the cluster handles everything else.

How replicas land somewhere

There is no scheduler. When a workload spec gossips out, every node independently scores itself as a candidate host: does it have spare capacity? Does it already have the artifact in its content store? Is it close to where calls are originating? Each node decides locally whether to claim a replica, and how many.

$ pln seed ./hello.wasm --min-replicas 2 The spec gossips out: spec(hello, hash=4f3a, target=2) Every node independently scores itself as a candidate host: Node A: cpu 20% cache hit score 0.85 -> claim Node B: cpu 80% cache miss score 0.30 -> skip Node C: cpu 30% cache hit score 0.70 -> claim Two replicas land. No node was told what to do.

The scoring system tries to minimise call latency by placing replicas near the traffic that uses them. As load shifts, replicas migrate. Kill a node and the survivors notice its claims have gone stale; the next scoring round produces a new placement and brings the workload back to its target replica count.

How calls find a replica

Every node knows where every replica lives. When you call a seed, the local node picks two candidates (power-of-two-choices) and routes to whichever responds with the lower load score. This works the same whether the caller is the CLI, a service, or another seed.

Seeds calling seeds

A seed can invoke another seed in two ways:

  • Host call: pollen_request("pln://seed/upper/handle", input) blocks until the inner seed returns. The runtime picks a replica, ships the call, and hands the response back. Good for composition where the outer seed needs the result.
  • Tail call: the seed returns a JSON marker {"kind":"tail_call","uri":"pln://seed/upper/handle","input":"..."}. The runtime releases the caller's WASM instance immediately, routes the inner seed, and returns its output to the original caller. Good for fan-out and policy gateways where the outer seed has nothing more to do.

The same model handles services: a seed can call pln://service/<name> to reach a TCP/UDP service somewhere on the mesh. Routing and authorisation are the same as for seed-to-seed calls.

Storage

Workload modules, static-site files, and named blobs are all just bytes. Pollen stores them in a single content-addressed store keyed by SHA-256, with each artifact's hash baked into the spec that references it.

$ pln seed ./big.bin payload Publisher writes the bytes to its local content store: CAS[publisher] += { 4f3a... -> bytes } And gossips a blob spec: blobspec(payload, hash=4f3a, publisher=...) Every other node now knows the blob exists. Any of them can stream it from any holder on demand, verifying the SHA-256 on arrival: Node B <------ stream over QUIC ------ Node A

Three things follow from content addressing:

  • Trust the bytes, not the source. Receivers verify the SHA-256 on arrival. A rogue peer cannot serve poisoned content under a known hash.
  • One-shot transfer. The same hash from any holder is the same bytes. Two peers can each pull from a third without coordination.
  • Free deduplication. Two seeds that share an artifact share one entry in the store.

Blobs are encrypted at rest with a per-node key. The plaintext only ever exists in memory, so a node's disk is safe to lose without leaking content.

Eviction

When a spec is unseeded, its hash becomes a tombstone in the gossip layer. Holders that don't see any other live spec referencing the hash drop it from their store on the next sweep. There is no explicit GC step.

Authorisation

Pollen's authorisation model is two parts: a chain of cryptographic capabilities that say who you are, and policy clauses on each spec that say who may use it. The runtime gate compares the two on every invoke, fetch, or connect.

The capability tree

Every cert in the cluster traces back to the root key created by pln init. Admin authority can be delegated; properties (key/value metadata) can be baked into a cert at issue time. Capability subsetting is enforced at issuance: a leaf cannot be issued more rights than the issuer holds.

[ root cert ] | | delegates admin authority | +-------------+-------------+ | | [ admin-A ] [ admin-B ] | | | delegates leaf certs | | [ leaf-A ] [ leaf-B ] role=engineer role=admin team=backend team=ops

The relationships are stored on the gossiped CRDT, so every node can independently verify any cert. Revocation works through a deny set on the same document; the CRDT propagates revocations the same way it propagates everything else.

Properties

Properties are key/value strings baked into a cert. Seeds see the caller's peer key and properties on every invocation, so authorisation, routing, and policy decisions can live inside workload code. Operators bake them in at issue time:

pln invite --prop role=engineer --prop team=backend
pln grant <peer> --prop tier=privileged
pln bootstrap ssh root@host --prop region=eu

The root node's own properties live in config.yaml and are re-applied on every pln up. Replace them with pln props.

The policy gate

Every published spec — workload, service, static site, named blob — can carry policy clauses. The runtime gate evaluates them against the caller's properties on every call.

spec(hello, --allow-props role=admin) On every call, the runtime gate runs: for each --allow-props clause on the spec: require caller's properties match exactly if any clause fails: close the stream silently So: leaf-A calls hello { role=engineer } -> deny (silent) leaf-B calls hello { role=admin } -> allow

Clauses are conjunctive: every --allow-props on the spec must match. To express disjunction ("editors or admins"), mint both groups with a shared property and require it on the spec:

pln grant <editor> --prop tier=privileged
pln grant <admin>  --prop tier=privileged
pln serve 8080 ops --allow-props tier=privileged

Two consequences of pushing disjunction up to the issuance side:

  • The runtime gate stays trivially fast. String equality across a few clauses, no expression evaluator on the hot path.
  • Policy is auditable. Who can call what is decided when you mint the cert, not at every invocation. The set of capable peers is visible from pln status nodes --wide.

Without any policy flag a published resource is open to every authenticated peer in the cluster. Failed gate checks close the stream silently — a denied caller can't probe the resource shape.

For the recipes that build on top of this, see Restrict who can call a workload and Run with the root machine offline.

How the pieces fit

A single pln seed exercises the whole stack:

  1. The CLI hashes the artifact and stores it in the local CAS.
  2. The supervisor signs a spec with the local admin cert; the spec carries the hash, any policy clauses, and the publisher's identity.
  3. The change goes into the CRDT and gossips to every peer.
  4. Each peer scores itself as a candidate host. Selected peers fetch the artifact from any holder over QUIC, verifying the hash on arrival.
  5. The first call into the workload routes to whichever replica is nearest and least loaded. The runtime gate checks the caller's properties against the spec's policy. If it passes, the call goes through.

No central coordinator was contacted. No shared secrets were used. Every step was decided locally against the same converged view of the world.

The other commands — pln serve, pln connect, pln invite, pln grant, pln deny — exercise the same machinery with different payloads. Once you've understood seeds, the rest of the surface follows.
