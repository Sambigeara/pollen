How-to

Goal-oriented recipes for common operator tasks. Each recipe is independent: pick the one that matches what you're trying to do. If you're new, run through the Quickstart first so the commands feel familiar.

Add a public relay

So peers behind NAT can find each other when direct UDP punching fails. A relay is just a normal cluster node that happens to be publicly reachable.

  1. Provision a public Linux box

    Anywhere with a public IPv4 address and a UDP port reachable from outside (default 60611). A small VM is plenty.

  2. Bootstrap it from your admin node

    pln bootstrap ssh relay-eu=root@198.51.100.10 --admin bootstrapping relay-eu=root@198.51.100.10... relay-eu=root@198.51.100.10 ok 1 succeeded, 0 failed

    --admin delegates admin authority too, so the relay can keep admitting peers if your root machine is offline. Use --publisher instead if the relay should be able to publish its own services or workloads but not admit peers, or drop both flags for a relay that only forwards traffic.

  3. Verify reachability

    pln status nodes PEERS NODE STATUS ADDR CPUs CPU MEM TUNNELS LATENCY IN OUT 7c9a4f8d (self) - - 8 2% 18% 1 - - - ab12cd34 online (public) 198.51.100.10:60611 4 1% 8% 1 12ms 0 0

    The (public) tag means the cluster has verified the node is reachable from outside, not just hinted. New invite tokens minted from any admin will include this address as a bootstrap target.

Related: Concepts → The mesh, pln bootstrap ssh, pln up --public.

Run with the root machine offline

So you can take your laptop with you. Delegate admin authority to a node that stays online, and the cluster keeps issuing certs, admitting peers, and renewing leases without you.

  1. Pick the node that should hold admin

    A relay is a natural fit (always-on and well-connected). Note its peer ID:

    pln status nodes --wide PEERS NODE STATUS ADDR 7c9a4f8d3e2b1c5a6d8e9f0a1b2c3d4e5f6a7b8c... (self) - - ab12cd34e5f6789a1b2c3d4e5f6789a1b2c3d4e5... online (public) 198.51.100.10:60611

  2. Grant admin to that peer

    pln grant ab12cd34 --admin certificate issued (admin) to ab12cd34

    The --admin flag mints a delegation cert with CanDelegate and CanAdmit. The new admin can issue further delegations and admit new peers without involving root.

  3. Confirm and shut root down

    pln down

    The cluster keeps running. Issue invites, grant new peers, and seed workloads from the relay using --host or a context (see below).

Bake the tier in at join time. If you're standing up a fresh relay, skip the post-hoc grant: pln bootstrap ssh root@host --admin or pln invite --admin --subject <peer-pub> mints the relay with admin capability from the start. Use --publisher for a relay that should publish but not delegate.

Related: Concepts → Authorisation, pln grant, pln invite.

Restrict who can call a workload or service

Bake properties into the callers' certs, attach --allow-prop clauses to the spec. The runtime gate checks them on every call. See Concepts → Authorisation for the model.

  1. Tag the callers

    For peers that already exist:

    pln grant ab12cd34 --prop role=lead --prop team=backend certificate issued (leaf) to ab12cd34

    For new peers, bake at join time:

    pln invite --prop role=lead --prop team=backend

  2. Publish the resource with policy

    pln serve 8080 internal-api --allow-prop team=backend pln seed ./hello.wasm greeter --allow-prop role=lead

    Repeat --allow-prop for additional clauses; all must match (conjunction).

  3. Verify denials are silent

    Try to call the resource from a peer that doesn't carry the required property. The call closes the stream without revealing why; this prevents probing the resource shape.

    pln call greeter run 'hi' Error: stream closed by peer

Need disjunction? Mint both groups with a shared property and require it on the spec:

pln grant <editor> --prop tier=privileged
pln grant <admin>  --prop tier=privileged
pln serve 8080 ops --allow-prop tier=privileged

Pollen pushes disjunction up to the issuance side so the runtime gate stays trivially fast. The property name itself is your conceptual group.

Related: Concepts → Authorisation, pln serve, pln seed.

Publish a static site across the cluster

Every node with static-http enabled claims the site and serves it locally; requests are routed by Host header.

  1. Enable the static HTTP listener on each node that should serve

    pln set static-http set static-http=:8080 (run `pln restart` to apply) pln restart

    Pass a port to override: pln set static-http :9000. Run on every node you want to serve from.

  2. Publish the directory from any publisher or admin node

    pln seed ./public my-site seeded my-site (12 files, manifest 3a8c7d2e)

    pln seed hashes every file into the local content store, signs a static-site spec, and gossips both. Each enabled node pulls the missing blobs and starts serving.

  3. Confirm rollout

    pln status static STATIC NAME MANIFEST REPLICAS LOCAL PUBLISHER my-site 3a8c7d2e 3/3 * 7c9a4f8d curl -H "Host: my-site" http://localhost:8080/

    3/3 means three replicas claimed against three eligible nodes. Re-running pln seed ./public my-site is idempotent: the spec gets a fresh manifest, old blobs are GC'd once nothing references them.

Related: Concepts → Storage, pln seed, pln set static-http.

Pin a workload's placement

Pollen places replicas based on local scoring. You can tilt the scoring with flags, or override it for special cases.

Run a single replica everywhere

Use when every node should hold a copy (PDPs, edge caches, agents):

pln seed ./agent.wasm --everywhere

Set a higher target replica count

Use when you want headroom across the cluster. Default is two:

pln seed ./api.wasm --min-replicas 5

Restrict by property

Tag the eligible nodes, then require the tag on the spec. The capability gate stops other nodes from claiming the spec at all:

pln grant <node> --prop region=eu pln grant <node> --prop region=eu pln seed ./eu-only.wasm --allow-prop region=eu

Strictly, --allow-prop gates callers, not placement. To restrict placement specifically you'd compose a policy that matches host-side properties; the simplest workaround for now is property-tagging the nodes you want to host and gating the resource so only they can act on it.

Related: Concepts → Workloads, pln seed.

Export a blob's contents to a local file

Blobs are encrypted at rest, so reading one from the on-disk store is not possible directly. pln fetch is the export path: it streams plaintext bytes from the publisher and writes them to the file you choose.

  1. Pick the destination

    The second positional argument is the destination. Use a file path to write to disk, or - to write to stdout (handy for piping into tar, jq, file, and so on).

    pln fetch payload ./payload.bin wrote 9f2b...e814 to ./payload.bin

  2. Verify

    Hash the file locally; it must match the blob's content address.

    shasum -a 256 ./payload.bin

The local content store is not modified by fetch. If you want this node to also host the blob, use pln seed separately.

Related: Concepts → Storage, pln fetch.

Manage multiple clusters from one shell

Contexts let you point your CLI at different daemons without re-typing flags. The reserved name default always resolves to the local daemon at $PLN_DIR.

  1. Add a context for the remote cluster

    pln ctx add prod root@prod.example.com --from default

    Targets containing @ bind to a remote daemon over SSH. --from default imports your local admin keypair so this shell can act as admin against the remote cluster.

  2. Switch active context

    pln ctx use prod pln status # now reads prod

    Override per-command via $PLN_CONTEXT=prod pln status, or skip contexts entirely with -H root@prod.example.com pln status.

  3. Switch back

    pln ctx use default

Local-only contexts. Targets that are absolute paths bind to a local pln dir, useful for running multiple test clusters on one machine. Each gets its own UDP port assigned automatically so they coexist with the default context.

Related: pln ctx.

Rotate a node's properties

Properties are baked into certs, so a rotation means re-issuing the cert. The flow differs for delegated peers and the root.

Delegated peer

Re-issue with pln grant. The new cert supersedes the old one and gossips out:

pln grant ab12cd34 --prop role=primary --prop region=eu certificate issued (leaf) to ab12cd34

Root node

Root issues its own cert from config.yaml, so update the persisted properties and let the next pln up re-issue:

pln props role=primary region=eu properties updated (run `pln restart` to apply) pln restart

Replace semantics: positional args define the complete set, so any unlisted key is removed. Use pln props --clear to wipe.

Related: Concepts → Authorisation, pln props, pln grant.

Revoke a peer

Tear down a peer's connections and put it on the deny set so it can't rejoin without a new cert.

  1. Identify the peer

    pln status nodes

  2. Deny it

    pln deny ab12cd34 denied ab12cd34

    The deny gossips out. Every node tears down its tunnels to the denied peer and refuses to admit any cert that chains through it.

  3. Verify

    pln status nodes offline peers: 1 (use --include-offline)

Cascading revocation. If the denied peer had delegated admin to others, those leaves' certs chain through it and become invalid too. pln status nodes --include-offline shows the affected peers.

Related: pln deny, Concepts → Authorisation.

Upgrade across the cluster

There is no rolling-upgrade orchestration. Run pln upgrade --restart on each machine independently.

pln upgrade --restart

Wraps Homebrew on macOS and the install script on Linux (preserving install method). --restart bounces the background service so the new binary picks up.

Related: pln upgrade.
