Quickstart

Install Pollen, stand up a single-node cluster, deploy a WASM workload, and call it. By the end you'll have a working mesh you can extend with a second machine, a service, or a static site without learning anything new.

What you need

  • A Mac or Linux machine. Pollen is pure Go, no CGO; a Raspberry Pi works fine.
  • UDP port 60611 reachable on the loopback interface. Outbound UDP for talking to other peers later.
  • Sudo on the machine for the daemon install (Linux only).

1. Install

The installer wraps Homebrew on macOS and the right package manager on Linux (apt, dnf, yum, zypper). Use it instead of grabbing a binary so upgrades and uninstalls are managed natively.

curl -fsSL https://pln.sh/install.sh | bash

On Linux this also creates a pln system user, provisions /var/lib/pln, and adds you to the pln group so the CLI can reach the daemon socket. Open a new shell so the group membership applies. Verify the install:

pln version pln 0.x.y (build 9eefda9, 2026-05-10)

macOS first run. The first time pln tries to reach a LAN peer, macOS will ask for Local Network access. Grant it. If you miss the prompt, see Troubleshoot → macOS Local Network.

2. Stand up a cluster

One command creates the root cluster on this machine. Pollen mints an admin keypair, writes a self-signed root cert, and persists state under ~/.pln (or /var/lib/pln on Linux when installed as a service).

pln init initialized root cluster root_pub: 7c9a4f8d3e2b1c5a6d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c cluster_id: a1b2c3d4e5f6789a1b2c3d4e5f6789a1b2c3d4e5f6789a1b2c3d4e5f6789a1b2

Now start the daemon. -d detaches into the background service (launchd on macOS, systemd on Linux). Skip -d to run in the foreground if you want to see the logs directly.

pln up -d Successfully started `pln` (label: homebrew.mxcl.pln)

Check that you have a one-node cluster:

pln status PEERS NODE STATUS ADDR CPUs CPU MEM TUNNELS LATENCY IN OUT 7c9a4f8d (self) - - 8 2% 18% 0 - - -

That's a fully functional Pollen cluster. Everything below works against this baseline; bringing in a second machine just adds another row to the table.

3. Deploy a workload

Pollen runs WebAssembly. Any module compiled to Extism's plugin shape works, which means Go, Rust, JavaScript, Python, C#, or Zig. We'll use the bundled echo example, which takes a string and returns it.

git clone https://github.com/sambigeara/pollen.git ~/pollen cd ~/pollen/examples/echo && ./build.sh building echo.wasm... building upper.wasm... done

pln seed publishes content into the cluster. The kind is autodetected from what you point at: a .wasm file becomes a workload, a directory becomes a static site, anything else becomes a blob.

pln seed ./echo.wasm seeded echo (4f3a7c91)

Pollen has hashed the artifact, signed a workload spec with your admin cert, and gossiped both into the cluster. With one node you'll see one replica. With many nodes, replicas land wherever there's capacity nearby.

pln status seeds SEEDS NAME HASH STATUS REPLICAS LOCAL UPTIME echo 4f3a7c91 ready 1/2 * 3s

1/2 means one active replica out of a target of two. The default --min-replicas is two; with only one node available the second replica waits for capacity to appear.

4. Call it

Invoke a function on the workload. Calls are routed to the nearest, least-loaded replica. With one node, that's local.

pln call echo echo 'hello world' hello world

That round-trip went through the local supervisor, into the WASM runtime, and back out. From a second node the same call would route over the mesh to whichever replica answered first.

Want to see seeds calling seeds? The echo module ships with a shout function that delegates to the upper seed via host call. Seed both, then call shout:

pln seed ./upper.wasm
pln call echo shout 'hello world'      # → HELLO WORLD

That second call hops through the mesh to wherever upper is running.

5. Extend the cluster

Adding a second machine gives you the rest of the story: replicas spread, calls take the shortest path, and the cluster keeps working when nodes leave. From any admin-capable node:

pln bootstrap ssh user@second-machine # or several at once, with labels: pln bootstrap ssh edge-eu=root@10.0.0.5 edge-us=root@10.0.0.6

Linux targets only, SSH as root or with passwordless sudo. The orchestrator installs Pollen on the target, mints a cert, and starts the daemon. Within seconds the new node appears in pln status and replicas start to balance.

Out of band joins. If you can't SSH directly, mint an invite token on the admin and ship it (Slack, email, anywhere). On the joiner, run pln join <token>. Tokens are signed and self-contained; the default TTL is five minutes.

What just happened

Three things, in layers:

  • A signed root. pln init created an ed25519 admin keypair and a self-signed root cert. Every cert in the cluster traces back to that root via a chain of delegations. Membership is cryptographic; there are no shared secrets to leak.
  • A converging document. Every node holds the same CRDT describing peers, services, workloads, sites, and blobs. pln seed writes a fact into that document; the gossip layer fans it out to every other node, which then independently decides what to do about it.
  • Local placement. No node tells any other node to host the workload. Each one scores itself against the spec — capacity, cached artifacts, proximity to traffic — and claims a replica if it's a good fit. Add a node, and it joins the scoring round.
