Troubleshoot

Common symptoms with their likely causes and the fix. Use the symptom you actually see as the lookup key. If nothing here matches, the daemon log is the next place to look — pln logs -f follows it live.

Before you dig in

Three commands will tell you most of what you need to know:

pln status # peers, services, seeds, sites, blobs at a glance pln logs -f -n 200 # tail the daemon log; ctrl-c to exit pln version # confirm the binary you have

If pln status errors immediately, the daemon isn't running. Skip to Daemon won't start or isn't reachable.

macOS: sendmsg: no route to host on LAN dials

Symptom. Calls to peers on the same network silently fail or surface EHOSTUNREACH. Internet peers work fine. Restarting the daemon doesn't help.

Cause. macOS Local Network Privacy is denying pln access to the LAN. The first time the binary tries to reach a LAN peer, macOS prompts; if you missed the prompt, or the binary's signature changed (after an upgrade or ad-hoc resign), the privilege is revoked silently.

Fix. Grant access in System Settings → Privacy & Security → Local Network, then restart the daemon:

pln restart

If pln isn't listed, force the prompt by running pln status while a LAN peer is up.

Linux: permission denied on the daemon socket

Symptom.

Error: dial unix /var/lib/pln/pln.sock: permission denied

Cause. The system daemon runs as the pln user and the socket is group-owned by pln. Your shell user isn't in the group yet.

Fix. The installer adds you to the group automatically when run with sudo. If something went sideways:

sudo usermod -aG pln "$USER"

Open a new shell for the change to take effect (or newgrp pln in the current one).

Daemon won't start or isn't reachable

Symptom.

Error: control plane unreachable: dial unix /var/lib/pln/pln.sock: connect: no such file or directory

or the daemon flips to failed in brew services / systemctl status pln.

Cause(s). One of:

  • Port already in use. Another process holds UDP 60611 (default).
  • State directory unwritable. Ownership of ~/.pln or /var/lib/pln got out of sync after a manual file copy or a failed install.
  • Cert expired in degraded mode. The daemon will start but log a warning and refuse some operations until renewal succeeds.

Fix. Check the log first:

pln logs -n 200

For port collisions, change the listening port: pln up --port 60612 (foreground) or edit the service unit. For ownership, run sudo chown -R pln:pln /var/lib/pln on Linux. For an expired cert, see Cert expired.

A peer shows offline that should be online

Symptom.

pln status nodes offline peers: 1 (use --include-offline)

Cause(s).

  • Daemon down on the peer. The mesh marks unreachable peers offline after a short grace period.
  • NAT punching failed and there's no relay. The two peers can't find a direct path and can't relay either.
  • Firewall. A new outbound rule blocks UDP 60611 in either direction.

Fix. SSH to the peer and check pln status there. If the daemon is up but isolated, add a relay (recipe) so peers behind NAT can fall back to it. For firewalls, allow outbound UDP on the configured port (default 60611) and any other ports the relay uses.

Seed never reaches its target replica count

Symptom.

pln status seeds SEEDS NAME HASH STATUS REPLICAS LOCAL UPTIME hello 4f3a7c91 pending 0/2 - 30s

Cause(s).

  • No node has capacity. The scoring round didn't find a candidate. Add a node, or raise the per-workload limits (--memory, --timeout) to fit on what you have.
  • Artifact hasn't replicated yet. A scheduled host is still pulling the WASM bytes. Watch pln status blobs to confirm the hash is reaching new nodes.
  • Spec policy excludes every node from acting on it. Rare, but possible if you used --allow-prop on a workload such that no node-side identity matches.

Fix. Watch the daemon log on the candidate hosts (pln logs -f on each). The supervisor logs scoring decisions and any failures (CAS pull errors, validation failures).

pln seed fails with publish denied

Symptom.

pln seed ./hello.wasm --allow-prop role=admin Error: publish denied: local cert is missing prop "role" (policy requires "admin")

Cause. Hosting a workload includes loopback invocation, so the spec's policy must hold against the host's own cert. The publisher is checked at pln seed time: if the local cert doesn't satisfy the policy, the spec is rejected up front rather than left orphaned in gossip.

Fix. Either remove the offending --allow-prop clause, or grant the local node the required property first:

pln grant $(pln id) --prop role=admin # from an admin shell pln seed ./hello.wasm --allow-prop role=admin

See Authorisation for why hosts are gated alongside callers.

pln fetch fails with no publisher known for blob

Symptom.

pln fetch ab12cd34 ./out.bin Error: no publisher known for blob ab12cd34

Cause. Fetch always streams from the blob's publisher. If the blob has no BlobSpec in gossip (an orphan, or a digest you typed wrong) there is no origin to talk to.

Fix. List the blobs the cluster knows about and pick a name or matching digest:

pln status

pln fetch fails with open stream to publisher … failed

Symptom. Fetch returns an error referencing the publisher's short peer ID.

Cause. The publisher is unreachable: offline, partitioned, or behind NAT that the mesh hasn't punched yet. Fetch does not fall back to other holders.

Fix. Confirm the publisher is online in pln status. If it stays offline, the bytes are not retrievable until it returns; pin a copy locally with pln seed in advance for content you need available without the origin.

Call returns stream closed by peer

Symptom.

pln call greeter run 'hi' Error: stream closed by peer

Cause. Authorisation denied. The runtime gate evaluates --allow-prop clauses against the caller's cert; failed matches close the stream silently to prevent probing.

Fix. Inspect the spec and the caller's cert:

pln status seeds --wide # shows policy on the spec pln status nodes --wide # shows your peer's properties

Then either add the missing property to your cert (pln grant <your-peer> --prop key=value from an admin shell) or relax the spec by re-publishing without the offending clause.

Cert expired

Symptom. Daemon starts but logs:

delegation certificate has expired — starting in degraded mode, will attempt renewal

Some operations still work (read-only status); others fail until a fresh cert arrives.

Cause. The peer was offline through its renewal window and the cluster's automatic renewal pipeline hasn't yet caught up. The default delegation TTL is 30 days.

Fix. If the peer can reach an admin-capable node, renewal is automatic — just wait a minute. If not, mint a fresh cert from any admin shell:

pln grant <peer-id> --prop key=value … # re-issue with whatever properties were in place

For root, restart: pln restart re-issues the self-signed cert from config.yaml.

pln bootstrap ssh fails

Symptom.

user@host failed (ssh: handshake failed) 0 succeeded, 1 failed hint: for failed hosts, use `pln invite` and run `pln join <token>` on the remote host

Cause(s).

  • SSH access not set up. Bootstrap needs SSH as root or with passwordless sudo.
  • Target isn't Linux. Bootstrap-ssh is Linux-only.
  • Network blocks SSH. Firewall or VPN.

Fix. Test SSH directly: ssh user@host sudo -n true should succeed silently. If it doesn't, fix that first. As the hint suggests, you can always fall back to the out-of-band path:

pln invite --subject $(ssh user@host pln id) ssh user@host pln join <token>

Static site returns 404

Symptom. A site you published responds 404 instead of serving content.

Cause(s).

  • Static-HTTP not enabled on the node you're hitting. Sites are routed by Host header on each node's static-http listener. A node without that listener can't serve sites.
  • Spec hasn't reached this node yet. Possible right after publish; usually a couple of seconds.
  • Wrong Host header. The site name needs to match what you used when publishing.

Fix.

pln set # confirm static-http is set pln status static --include-offline curl -H "Host: my-site" -v http://localhost:8080/

If static-http isn't set, run pln set static-http and pln restart.

Properties don't seem to take effect

Symptom. You ran pln props or pln grant --prop but downstream policy decisions still use the old values.

Cause. Properties live in certs, so a change is a cert re-issue. Until the new cert reaches the relevant party, the old cert is what the gate sees.

  • For root: pln props updates the persisted file but doesn't re-issue the cert in-flight; restart to apply (pln restart).
  • For a delegated peer: pln grant issues immediately, but the cert needs to gossip to whichever node is enforcing the policy. Usually a couple of seconds.

Fix. Restart root after pln props. For delegated peers, watch pln status nodes --wide until the new properties show on the affected node.

Last resort: nuke and start over

If a node's local state is wedged beyond reason and you don't mind losing it:

pln down pln purge --yes # wipes cluster state, preserves identity keys pln purge --yes --include-keys # wipes everything, including identity

Then re-init or re-join. --include-keys means the node will appear as a fresh identity to the cluster, so previous peers will refuse to talk to it without a fresh invite.

Still stuck

Open an issue with the symptom, the steps that produced it, and the relevant log excerpt at github.com/sambigeara/pollen/issues. Pollen is small enough that most issues get a fix or a workaround quickly.
