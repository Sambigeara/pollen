Pollen runs a zero-trust mesh and a WASM workload runtime out of one static binary. These pages cover install, getting a cluster up, deploying workloads, and exposing services.

Install

curl -fsSL https://pln.sh/install.sh | bash

Wraps Homebrew on macOS and apt/dnf/yum/zypper on Linux. Linux package-manager selection is based on /etc/os-release; unknown distros are refused instead of guessed, and --method tarball opts in to a /usr/local/bin binary install with the same daemon provisioning. On macOS, state lives in ~/.pln. On Linux, the installer runs pln daemon install to provision /var/lib/pln under a dedicated pln system user and adds the invoking user to the pln group for CLI access. Override either default with $PLN_DIR.

One-node cluster

pln init
pln up

That's a single-node cluster, fully functional. Add -d to up to detach into the background service. Everything below works against this baseline.

Add a node

Two paths. Pick whichever fits your access model.

SSH (zero-touch)

Installs Pollen, joins the cluster, and starts the daemon. Linux targets only. Needs SSH as root or with passwordless sudo.

pln bootstrap ssh user@host
pln bootstrap ssh relay-eu=root@10.0.0.5 --admin

Prefix a target with name= to label the node. Pass --admin to delegate admin authority, so the cluster keeps issuing certs and admitting peers when your root machine is offline.

Invite token (out of band)

Mint on an admin, ship the token over any channel, redeem on the joiner.

# on the admin
pln invite                       # any joiner
pln invite --subject <peer-pub>  # bound to one node
# on the joiner
pln join <token>

Tokens are signed and self-contained, carrying the cluster root key and bootstrap addresses. Pass --subject with the recipient's pln id to bind a token to one node; omit it for an open invite. Safe to send over Slack or email. TTL defaults to 5 minutes.

Run a workload

pln seed ./hello.wasm
pln call hello greet '{"name":"world"}'

pln seed publishes content. The kind is autodetected from the source: a .wasm file becomes a workload, a directory becomes a static site, anything else becomes a blob. Pass an optional name as the second arg.

Replicas land on whichever nodes have capacity and proximity. There's no scheduler; every node decides locally. Tune with --min-replicas, --everywhere, --memory, --latency-slo.

Authoring. Workloads are Extism plugins. Author them in Go, Rust, JavaScript, Python, C#, Zig, and more. A workload invokes another seed by host call (pln://seed/<name>/<fn>) or by returning a tail_call marker, so authz, routing, and policy can live inside WASM.

Expose a service

# machine A
pln serve 8080 api
# machine B
pln connect api
curl localhost:8080

TCP or UDP (--udp), with mTLS end-to-end and NAT traversal built in. Connections punch directly when both sides can reach each other; otherwise they relay over the shortest mesh path. Persisted to config.yaml so they re-establish on restart.

Publish a site

pln seed ./public my-site

Walks the directory, hashes every file, and gossips them. Every node with static-http enabled claims the site and serves it on :8080 (routed by Host header). Override the port with pln set static-http :9000.

curl -H "Host: my-site" http://<node-addr>:8080/

Re-running the same command is idempotent: it supersedes the existing site with a fresh manifest. Old blobs are GC'd once no spec references them.

Share a blob

pln seed ./big.bin payload   # named
pln seed ./big.bin           # anon, hash-only
pln fetch payload            # pull onto this node
pln unseed payload           # drop the name + bytes

Blobs are content-addressed bytes. Streamed peer-to-peer over QUIC, gossip-advertised, and verified on arrival by SHA-256. Anonymous blobs print just their hash; named blobs are also resolvable by name across the cluster.

Day two

  • pln status shows peers, services, seeds, sites, and blobs. Use --wide for full IDs and metrics, --include-offline to see departed peers and orphaned blobs.
  • pln logs -f tails the background daemon's log.
  • pln daemon upgrade --restart pulls the latest binary and bounces the service.
  • pln set changes daemon config: http, static-http, control-addr, log-level. Run with no args to list keys; restart to rebind listeners.
  • pln context switches your admin shell between clusters or remote hosts.
  • pln deny <peer-id> revokes a peer's membership and tears down its connections.
  • pln down and pln restart stop or restart the local node.

Authorisation

Every link is mTLS. On top of that, one gate is wired today: service_connect, fired when a peer opens a tunnel to a service on this node (matched by service name, not port). It shipped first because a Service is defined in the local config.yaml and terminates on its owning node — every fact the gate needs is right there. Gates for the other resource types — blobs, workloads, specs — are next, and they're the harder case: those resources aren't pinned to a single node, so the decision has to hold cluster-wide rather than locally on the node that owns the resource.

The gate sees two property bags: the caller's delegation-cert attributes (subject) and the service publisher's properties (resource). It hands them to whichever evaluator you've bound — either the built-in YAML attribute_matcher, or a WASM seed in the cluster acting as a policy decision point.

Subject properties

Bake attributes into a delegation cert at issue time. They flow to the gate as subject.properties and are visible to workloads at call time.

pln grant <peer-id> --prop role=lead --prop team=backend
pln invite --prop tier=gold      # bake at join time

The root node sets its own attributes through configuration rather than self-grant. Bake them in at pln init, or edit config.yaml and restart:

pln init --prop role=primary --prop region=eu
# config.yaml
properties:
  role: primary
  region: eu

On every pln up, root re-issues its self-signed cert from config.yaml if the attributes have drifted, and the cluster's regular renewal pipeline keeps the cert fresh from then on. Delegated peers continue to receive properties imperatively via pln grant or pln invite — root is asymmetric because it has no upstream issuer.

Service properties

Attach properties to a service when you serve it. They flow to the gate as resource.properties.

pln serve 8080 internal --prop '{"public":true}'
String vs. typed values. --prop k=v always parses v as a string. For booleans and numbers, use the JSON form: --prop '{"public":true}'.

Match against rules

The built-in attribute_matcher evaluates YAML rules against dot-separated paths into the request. First match wins; no match falls through to deny. Scalar values match by equality; list values are any-of.

# /etc/pln/rules.yaml — referenced by evaluator.matcherRules in config.yaml.
rules:
  - match:
      action.name: connect
      resource.properties.public: true
    decision: allow

  - match: {}
    decision: deny
    reason: default deny

Bind the gate in config.yaml; restart the daemon to apply:

evaluator:
  matcherRules: /etc/pln/rules.yaml
  gates:
    service_connect: attribute_matcher

Rules reload on SIGHUP — edit, send the signal, no restart. Malformed files leave the previous rules in place. A reference ruleset lives at examples/matcher/rules.yaml.

Request envelope

The matcher sees an AuthZen-shaped envelope, populated from the in-flight connect:

{
  "subject":  {"type": "peer",    "id": "<pubkey>",       "properties": {"role": "lead", "team": "backend"}},
  "action":   {"name": "connect"},
  "resource": {"type": "service", "id": "internal",        "properties": {"public": true}}
}
