Pollen runs a zero-trust mesh and a WASM workload runtime out of one static binary. These pages cover install, getting a cluster up, deploying workloads, and exposing services. For internals, see ARCHITECTURE.md.

Install

curl -fsSL https://pln.sh/install.sh | bash

Wraps Homebrew on macOS and apt/yum on Linux. On macOS, state lives in ~/.pln. On Linux, the installer runs pln daemon install to provision /var/lib/pln under a dedicated pln system user and adds the invoking user to the pln group for CLI access. Override either default with $PLN_DIR.

One-node cluster

pln init
pln up

That's a single-node cluster, fully functional. Add -d to up to detach into the background service. Everything below works against this baseline.

Add a node

Two paths. Pick whichever fits your access model.

SSH (zero-touch)

Installs Pollen, joins the cluster, and starts the daemon. Linux targets only. Needs SSH as root or with passwordless sudo.

pln bootstrap ssh user@host
pln bootstrap ssh relay-eu=root@10.0.0.5 --admin

Prefix a target with name= to label the node. Pass --admin to delegate admin authority, so the cluster keeps issuing certs and admitting peers when your root machine is offline.

Invite token (out of band)

Mint on an admin, ship the token over any channel, redeem on the joiner.

# on the admin
pln invite                       # any joiner
pln invite --subject <peer-pub>  # bound to one node
# on the joiner
pln join <token>

Tokens are signed and self-contained, carrying the cluster root key and bootstrap addresses. Pass --subject with the recipient's pln id to bind a token to one node; omit it for an open invite. Safe to send over Slack or email. TTL defaults to 5 minutes.

Run a workload

pln seed ./hello.wasm
pln call hello greet '{"name":"world"}'

pln seed publishes content. The kind is autodetected from the source: a .wasm file becomes a workload, a directory becomes a static site, anything else becomes a blob. Pass an optional name as the second arg.

Replicas land on whichever nodes have capacity and proximity. There's no scheduler; every node decides locally. Tune with --min-replicas, --everywhere, --memory, --latency-slo.

Authoring. Workloads are Extism plugins. Author them in Go, Rust, JavaScript, Python, C#, Zig, and more. One host call lets a workload invoke another (seed/<name>/<fn>), so authz, routing, and policy can live inside WASM.

Expose a service

# machine A
pln serve 8080 api
# machine B
pln connect api
curl localhost:8080

TCP or UDP (--udp), with mTLS end-to-end and NAT traversal built in. Connections punch directly when both sides can reach each other; otherwise they relay over the shortest mesh path. Persisted to config.yaml so they re-establish on restart.

Publish a site

pln seed ./public my-site

Walks the directory, hashes every file, and gossips them. Other nodes claim the site according to --min-replicas and serve it on :8080 (routed by Host header).

curl -H "Host: my-site" http://<node-addr>:8080/

Re-running the same command is idempotent: it supersedes the existing site with a fresh manifest. Old blobs are GC'd once no spec references them.

Share a blob

pln seed ./big.bin payload   # named
pln seed ./big.bin           # anon, hash-only
pln fetch payload            # pull onto this node
pln unseed payload           # drop the name + bytes

Blobs are content-addressed bytes. Streamed peer-to-peer over QUIC, gossip-advertised, and verified on arrival by SHA-256. Anonymous blobs print just their hash; named blobs are also resolvable by name across the cluster.

Day two

  • pln status shows peers, services, seeds, sites, and blobs. Use --wide for full IDs, --include-offline to see departed peers.
  • pln logs -f tails the background daemon's log.
  • pln daemon upgrade --restart pulls the latest binary and bounces the service.
  • pln set changes daemon config like HTTP bind or log level. Run with no args to list keys.
  • pln context switches your admin shell between clusters or remote hosts.
  • pln down and pln restart stop or restart the local node.

Authorisation

Every link is mTLS. On top of that, two extension points let you decide who can do what: cert properties baked into each issued cert flow into authorisation requests, and a gate router dispatches each protected operation through the evaluator of your choice — built-in YAML matcher or a WASM PDP.

Cert properties

Properties baked into a delegation cert are visible to seeds at call time, so admission, routing, and policy can run inside WASM without sharing secrets.

pln grant <peer-id> --prop role=lead --prop team=backend
pln invite --prop tier=gold      # bake at join time

Gates

Every protected operation runs through a named gate. Each gate independently binds to one evaluator; unbound gates fall through to the configured default.

  • blob_fetch — peer-to-peer blob downloads.
  • workload_call — incoming WASM invocations.
  • service_connect — inbound tunnel connects (by service name, not port).
  • spec_publish, grant_issue, seed_placement — reserved for forthcoming hooks.

Bind gates in config.yaml; restart the daemon to apply:

evaluator:
  default: allow_all                # fall-through for unbound gates
  gates:
    blob_fetch: attribute_matcher
    workload_call: seed/cerbos-pdp
  matcherRules: /etc/pln/rules.yaml # required if any gate uses attribute_matcher

Authorisation request

Both evaluators see the same AuthZen-shaped envelope. The router populates it from the in-flight operation:

{
  "subject":  {"type": "peer", "id": "<pubkey>", "properties": {"role": "lead", "team": "backend"}},
  "action":   {"name": "fetch"},
  "resource": {"type": "blob", "id": "<hash>",   "properties": {"public": true}}
}

subject.type is peer for direct callers (properties = the peer's delegation-cert attributes) or seed when a workload calls another via OnBehalfOf (properties = the publisher's claim). resource.properties are the publisher's --prop values verbatim.

Attribute matcher

YAML rules that match on dot-separated paths into the request. First rule whose match clause holds wins; no match falls through to deny. Scalar values match by equality; list values are any-of. Suitable for simple role-based and tag-based policy.

# /etc/pln/rules.yaml — referenced by evaluator.matcherRules in config.yaml.
rules:
  - match:
      subject.properties.role: admin
    decision: allow
    reason: admins have full access

  - match:
      action.name: fetch
      resource.properties.public: true
    decision: allow

  - match:
      action.name: connect
      resource.id: [api, internal-api]
      subject.properties.team: backend
    decision: allow

  - match: {}
    decision: deny
    reason: default deny

Rules reload on SIGHUP — edit, send the signal, no restart. Malformed files leave the previous rules in place. The matcher is opt-out of decision caching; expensive evaluators (PDP seeds, remote services) keep the default 1s cache.

PDP seed

For policy that outgrows YAML — derived roles, conditions, query plans — point a gate at a WASM seed. The router dispatches each decision over seed/<name>/check with the request envelope as JSON input; the seed returns a JSON decision. A circuit breaker isolates a misbehaving PDP after five consecutive failures.

# config.yaml
evaluator:
  gates:
    blob_fetch: seed/cerbos-pdp
    workload_call: seed/cerbos-pdp

The PDP is a regular Pollen workload, deployed the same way as any other:

pln seed cerbos-pdp.wasm --everywhere   # local on every node — low gate latency

Implement a single export named check that consumes the JSON request and returns a JSON decision:

{"decision": true,  "context": {"reason_admin": "EFFECT_ALLOW"}}
{"decision": false, "context": {"reason_user": "team mismatch"}}

context.reason_user is surfaced to the denied caller; context.reason_admin is for operator logs. A worked example using Cerbos lives at examples/cerbos — it serves both application authz (a workload calling seed/cerbos-pdp/check from a protected endpoint) and cluster gates (the same WASM bound to gates.blob_fetch) from one policy bundle.

Inspect a gate

Dry-run a request through the live router without any side effect. Useful for confirming a policy edit before traffic hits it, and for debugging unexpected denies.

pln policy check workload_call \
  --subject peer:a9b8... --subject-prop role=lead \
  --action call \
  --resource seed:deadbeef... --resource-prop tier=gold
