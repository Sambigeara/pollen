# Pollen Demo

A global Pollen mesh with a pre-built Grafana dashboard, driven from `just`
recipes. 10 AWS nodes across Virginia, Oregon, Ireland, Stockholm, Frankfurt,
Tokyo, Singapore, Sydney, Mumbai, and São Paulo — plus a local root node on
your laptop.

## Prerequisites

- AWS credentials configured (`aws configure` or env vars)
- Terraform, Docker, and `just` installed
- SSH key at `~/.ssh/id_ed25519`

## Quick Start

```bash
cd infra
just demo-up             # build → deploy → start → bootstrap → seed (idempotent, ~5 min)
```

Open Grafana at <http://localhost:3000>.

The individual steps (all runnable in isolation) are:

```bash
just build-demo          # compile binaries + WASM + fetch world map
just deploy-demo         # terraform + push binary + provision 10 nodes + 1 exerciser host
just start-demo          # local root + store + prometheus + grafana
just bootstrap-demo      # form the 11-node cluster
just seed-demo           # deploy ingest → processor chain
```

## Load generation

Load is generated by a dedicated AWS EC2 host (provisioned by `deploy-demo`),
not by the pollen nodes themselves. Each target gets its own systemd instance
so multiple regions can be hit simultaneously and independently.

```bash
just start-exerciser usw 100       # 100 r/s at the usw pollen node, open-ended
just start-exerciser usw 200       # replaces the usw run at 200 r/s (idempotent)
just start-exerciser tok 50 5m     # 50 r/s at tok for 5 minutes, auto-stops
just exerciser-log usw             # tail journalctl for the usw instance
just stop-exerciser usw            # stop load to usw
just ssh-exerciser                 # SSH into the exerciser host
```

Each call traverses the full chain (ingest → processor → store). Sustainable
throughput depends on cross-mesh latency and the pollen node's per-workload
concurrency gate (which adapts to observed dial behaviour).

## Inspect Store Writes

SQLite lives in the `demo_store-data` Docker volume. Query it via a disposable
sidecar (no mount conflict with the running store):

```bash
# writes per second over the last 10s
docker run --rm -v demo_store-data:/data keinos/sqlite3 \
  sqlite3 /data/pollen-demo-store.db \
  "SELECT COUNT(*)/10.0 AS rps FROM events WHERE ts > strftime('%s','now')-10;"

# total rows
docker run --rm -v demo_store-data:/data keinos/sqlite3 \
  sqlite3 /data/pollen-demo-store.db "SELECT COUNT(*) FROM events;"
```

Wrap either in `watch -n1 '…'` for a live rate meter.

## Tear Down

```bash
just destroy-demo             # stops local stack + terraform destroy (all AWS infra)
```

Or step-by-step:

```bash
just stop-exerciser usw       # if running
just stop-exerciser tok       # if running
just stop-demo                # stop local root + docker stack
just destroy-demo             # terraform destroy + cleanup
```

## Architecture

```
laptop (root)
├── pln --dir /tmp/pln-demo up --port 60612  (http :9090 via `pln set`)
├── store (Docker, host network, :8080)
├── Prometheus (Docker, scrapes localhost:9090)
└── Grafana (Docker, :3000, ECharts plugin)

10× AWS c7g.medium (ARM64) — one per region
└── pln (systemd, dev binary pushed at deploy time, TCP control on :50051)

1× AWS t4g.small (us-east-1) — dedicated load generator
├── exerciser (systemd template: pollen-exerciser@<target>.service)
└── Prometheus metrics endpoints (:9192-:9201) scraped from laptop
```

Compute isolation: load generation never shares CPU with the pollen node
under test. The exerciser host talks to each pollen node via its TCP control
endpoint (protected by a shared-secret token persisted on each node at
`/var/lib/pln/control.token`).

Port 60612 avoids conflicts with any existing cluster on the default 60611.
