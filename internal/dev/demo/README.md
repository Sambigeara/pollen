# Pollen Demo

A global Pollen mesh with a pre-built Grafana dashboard. 10 AWS pollen nodes
across Virginia, Oregon, Ireland, Stockholm, Frankfurt, Tokyo, Singapore,
Sydney, Mumbai, and São Paulo, plus a dedicated load-generator host. The
demo narrative (`SCRIPT.md`) runs from a laptop: `pln init`, `pln bootstrap
ssh`, `pln seed`, `pln status`.

## Prerequisites

- AWS credentials configured (`aws configure` or env vars)
- Terraform, Docker, and `just` installed
- `pln` on PATH (`brew install pln` is fine)
- SSH key at `~/.ssh/id_ed25519`

## Infrastructure lifecycle

```bash
# One-time (per demo run)
just internal/dev/build-demo          # exerciser binary + WASM modules + world map
just internal/dev/deploy-demo         # terraform + authorized_keys + exerciser host (pollen nodes untouched)

# Per-recording
just internal/dev/start-observability # grafana + prometheus + fresh file-SD targets

# Drive the cluster from the laptop — see SCRIPT.md for the narrative
pln init
cat ~/bootstrap_peers.txt | pln bootstrap ssh -
pln seed …
pln status

# Load (optional, visible in Grafana)
just internal/dev/start-exerciser usw 1000 5m
just internal/dev/start-exerciser usw,use,eun,sao,tok 1000 5m
just internal/dev/stop-exerciser usw

# Teardown
just internal/dev/destroy-demo        # docker stack down + terraform destroy
pln ctx rm demo          # if the demo was run in a named context
```

The pollen nodes carry no pollen state before `pln bootstrap ssh` runs.
The bootstrap does the binary install via the release curl script,
enrols each node, starts the daemon, and binds the token-authenticated
control API (`:50051`) so the cluster is remotely manageable.

## Load generation

Load is generated by a dedicated AWS EC2 host (provisioned by
`deploy-demo`), not by the pollen nodes themselves. Each target gets its
own systemd instance so multiple regions can be hit simultaneously and
independently.

```bash
just internal/dev/start-exerciser usw 100       # 100 r/s at the usw pollen node, open-ended
just internal/dev/start-exerciser usw 200       # replaces the usw run at 200 r/s (idempotent)
just internal/dev/start-exerciser tok 50 5m     # 50 r/s at tok for 5 minutes, auto-stops
just internal/dev/exerciser-log usw             # tail journalctl for the usw instance
just internal/dev/stop-exerciser usw            # stop load to usw
just internal/dev/ssh-exerciser                 # SSH into the exerciser host
```

Each call traverses the full chain (ingest → terminal by default, or
ingest → processor → store when targeting `pln://seed/ingest/handle`).
Sustainable throughput depends on cross-mesh latency and the pollen node's
per-workload concurrency gate.

The exerciser talks to each pollen node over its TCP control endpoint
(bound by default at bootstrap time — see `pln bootstrap ssh --help`),
authenticated with a shared-secret token persisted at
`/var/lib/pln/control.token` on each node.

## Dashboard

| Panel | What |
|-------|------|
| **Mesh Topology** | ECharts world map. Blue = ingest, green = processor. Dot size = workload pressure. |
| **Nodes / Workloads / Replicas** | Cluster-wide counts |
| **Workload Pressure** | Demand-to-capacity ratio per workload over time |
| **Outbound Traffic** | Gossip byte rate from the root node |
| **Workload Placement** | Which workloads run on which nodes |

## Inspect store writes

If the demo used the `ingest/handle` chain (not the default firehose), the
store's SQLite lives in the `demo_store-data` Docker volume. Query via a
disposable sidecar:

```bash
docker run --rm -v demo_store-data:/data keinos/sqlite3 \
  sqlite3 /data/pollen-demo-store.db \
  "SELECT COUNT(*)/10.0 AS rps FROM events WHERE ts > strftime('%s','now')-10;"
```

|------|---------|

| 9091 | Prometheus (for debugging) |

| 50051/tcp | Per-node pollen control API (on remote nodes) |
| 60611/udp | Pollen mesh (remote nodes) |
| 60612/udp | Pollen mesh (laptop root, avoids conflict with any default-port cluster) |
